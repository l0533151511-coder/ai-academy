## ביקורת: האם כל מודול תורם ל-AtlasDesk?

AtlasDesk הוא הקפסטון המתמשך. לא כל מודול *צריך* לגעת בו — שלושת הטראקים הראשונים הם **תשתית-ידע
הכרחית** (מחשב, רשת, טרמינל, git, JS/TS, יסודות ML). לכפות עליהם AtlasDesk היה מלאכותי. AtlasDesk
מתחיל נכון ב-prompt-engineering, ומשם **כל מודול מוסיף לו יכולת אמיתית**:

| טראק | תרומה ל-AtlasDesk | ערך עסקי |
|------|-------------------|----------|
| prompt-ai-dev | מנוע שיחה (system prompt, זהות) | הבוט מדבר |
| ai-integration | tool calling, פלט-מובנה, semantic search, RAG | הבוט פועל ויודע |
| ai-agents | סוכן, multi-agent escalation, webhook | הבוט אוטונומי ומסלים |
| production-ai | ניטור, אבטחה, safety, streaming, testing | הבוט מוכן-לפרודקשן |
| saas-capstone | personas, תמחור, multi-tenancy, בקרת-גישה | הבוט מוצר מסחרי |

**פסק-דין:** אין מודול (מ-prompt-engineering ואילך) שלא מוסיף ל-AtlasDesk ערך עסקי אמיתי. אין צורך
בעיצוב-מחדש של מודול. הטראקים הפרה-AtlasDesk נכונים כתשתית ולא צריכים לגעת בו.

## סולם הפרויקטים (Difficulty Ladder)

מחשב-צעצוע → מפת-HTTP → משימת-טרמינל → פורטפוליו-git → api-caller → מסווג-ספאם → מזהה-ספרות →
מחשבון-טוקנים → **[AtlasDesk מתחיל]** מנוע-שיחה → ארכיטקטורת-פיצ'ר → פיצ'ר-TDD → הרחבה-בטוחה →
ספריית-פרומפטים → כלי-אמיתי → חיפוש-סמנטי → RAG → סוכן → multi-agent → webhook → ניטור → הקשחת-אבטחה
→ runbook-תקריות → מסמך-MVP → בקרת-גישה → **קפסטון production-ready** → סינתזת-האקדמיה.

הקושי עולה בהתמדה: מ"כלי בודד" בדפדפן, דרך "פיצ'ר מאחורי טסטים", ועד "מערכת multi-agent מנוטרת
ומאובטחת בפרודקשן". כל פרויקט מניח את הקודם.

## מה נשאר לבוגר לרכוש בעבודה (הגבולות הכנים)

אף אקדמיה לא מחליפה ותק. בוגר עדיין ירכוש בעבודה: קנה-מידה של מיליוני משתמשים, on-call אמיתי,
פוליטיקה ארגונית, ובעלות ארוכת-טווח על מערכת. האקדמיה נותנת את ה**שיפוט וההרגלים** שמאפשרים
לרכוש את אלה מהר — לא אשליה שהם כבר קיימים.
